# قواعد الواجهات وتجربة المستخدم والأداء (UI/UX & Performance Rules)

> [!IMPORTANT]
> هذا الملف جزء من بروتوكول الإدارة والتطوير الحاكم للوكيل. المرجع الأعلى والسلطة النهائية: `master_rules.md`

## 17. معايير متناغمة لطبيعة المشروع والواجهات المتجاوبة (Project Type & Responsiveness)
* يجب على الوكيل تحديد "نوع المشروع التقني" بدقة داخل ملف `project_map.md` فور انتهاء العصف الذهني (مثال: Web App / Mobile App / Backend Only).
* بناءً على نوع المشروع المعتمد، يلتزم الوكيل ومعمارية شبكة المراجعين بالمعايير التالية حتماً دون طلب إضافي من المستخدم:
  1. **تطبيقات الويب (Web Apps):** يُلزم الوكيل بكتابة كود متجاوب بالكامل (Responsive) يدعم الشاشات الثلاث (كمبيوتر، آيباد، جوال) في نفس كتلة الكود، باستخدام أسلوب **"التصميم أولاً للجوال" (Mobile-First Approach)** واعتماد شبكات مرنة (Flexbox / Grid) أو مكتبات التنسيق الذكية (مثل Tailwind CSS بوعي الـ Breakpoints `sm:`, `md:`, `lg:`).
  2. **تطبيقات الجوال الناتيف (React Native / Flutter):** يُلزم الوكيل بمراعاة هوامش الأمان للشاشات المختلفة (Safe Area Views)، وأبعاد الشاشات المتنوعة لنظامي iOS و Android.
  3. **المشاريع الخلفية (Backend Only):** في حال كان المشروع عبارة عن قاعدة بيانات أو API فقط، يتم **إيقاف وتعطيل "مُراجع تجربة المستخدم UI/UX Critic" تلقائياً** لتوفير استهلاك التوكنز، ويتحول وزنه التقييمي إلى مُراجع الأمن والأداء.
* يُحظر على مراجع الـ UI/UX تمرير أي كود واجهات ما لم يتم التحقق عبر أداة `/browser` المدمجة من سلامة تصميمه وتجاوبه على مختلف قياسات الشاشات ودعمه الكامل لاتجاه القراءة العربي (RTL).

## 24. معايير الوصولية الإلزامية (Accessibility — a11y)
* يُلزم الوكيل بتطبيق معايير الوصولية **WCAG 2.1 مستوى AA** كحد أدنى في جميع تطبيقات الويب والواجهات الأمامية:
  1. **تسميات ARIA (ARIA Labels):** يجب إضافة تسميات `aria-label` و `aria-describedby` و `role` المناسبة لجميع العناصر التفاعلية (أزرار، حقول إدخال، قوائم منسدلة، نماذج) لضمان توافقها مع قارئات الشاشة.
  2. **التنقل بلوحة المفاتيح (Keyboard Navigation):** يجب أن تكون جميع العناصر التفاعلية قابلة للوصول والتشغيل عبر لوحة المفاتيح فقط (Tab، Enter، Escape، أسهم الاتجاهات) مع ترتيب `tabindex` منطقي ومرئي.
  3. **نسب التباين اللوني (Contrast Ratios):** يجب ألا تقل نسبة التباين بين النص والخلفية عن **4.5:1** للنصوص العادية و **3:1** للنصوص الكبيرة، وفقاً لمعايير WCAG AA.
  4. **الـ HTML الدلالي (Semantic HTML):** يُلزم الوكيل باستخدام العناصر الدلالية الصحيحة (`<nav>`, `<main>`, `<article>`, `<section>`, `<header>`, `<footer>`, `<button>`, `<label>`) بدلاً من عناصر `<div>` و `<span>` العامة لتحسين البنية وتسهيل الوصول.
* يُحظر على مراجع الـ UI/UX تمرير أي كود واجهات لا يستوفي هذه المعايير الأربعة.

## 25. التوثيق التلقائي الإلزامي (Automatic Documentation)
* يُلزم الوكيل بإنشاء وصيانة التوثيق التقني التالي تلقائياً كجزء لا يتجزأ من عملية التطوير:
  1. **توثيق واجهات الـ API (API Documentation):** كل نقطة نهاية (Endpoint) يتم إنشاؤها يجب توثيقها باستخدام تعليقات **JSDoc** المنظمة (للمشاريع المبنية بـ JavaScript/TypeScript) أو مواصفات **OpenAPI/Swagger** (للمشاريع التي تحتوي على REST API)، مع تحديد: المسار، الطريقة (Method)، المدخلات المطلوبة والاختيارية، المخرجات المتوقعة، وأكواد الأخطاء المحتملة.
  2. **ملف README التقني:** يُلزم الوكيل بإنشاء ملف `README.md` شامل في جذر المشروع يحتوي على: وصف مختصر للمشروع، متطلبات التشغيل (Prerequisites)، خطوات التثبيت والتشغيل المحلي، البنية المعمارية للملفات (File Structure)، ومتغيرات البيئة المطلوبة (دون قيمها الحقيقية).
  3. **تحديث مستمر:** يتم تحديث التوثيق تلقائياً عند كل تعديل بنيوي أو إضافة ميزة جديدة تؤثر على الـ API أو طريقة التشغيل.

## 26. استراتيجية الأداء وتحسين الحمل (Performance Strategy & Optimization)
* يُلزم الوكيل بتطبيق استراتيجيات تحسين الأداء التالية عند الحاجة وبحسب طبيعة المشروع:
  1. **التحميل الكسول (Lazy Loading):** يجب تأجيل تحميل المكونات والصفحات والصور غير المرئية حتى لحظة الحاجة إليها، باستخدام `React.lazy()` أو `dynamic import()` أو ما يعادلها في الإطار المستخدم.
  2. **تقسيم الكود (Code Splitting):** يُحظر تجميع كامل تطبيق الويب في حزمة واحدة (Single Bundle) إذا تجاوز حجمها 250KB. يُلزم الوكيل بتقسيم الكود إلى حزم مستقلة لكل مسار (Route-based Splitting) أو لكل ميزة (Feature-based Splitting).
  3. **التخزين المؤقت (Caching):** يجب تطبيق استراتيجيات التخزين المؤقت المناسبة: HTTP Cache Headers للموارد الثابتة، وService Workers للتطبيقات التي تتطلب عمل دون اتصال، وتخزين مؤقت لاستجابات الـ API المتكررة عند الحاجة.
  4. **تحسين حجم الحزمة (Bundle Optimization):** يُلزم الوكيل بمراقبة حجم الحزمة النهائية واستخدام أدوات التحليل (مثل `webpack-bundle-analyzer` أو ما يعادلها) لاكتشاف وإزالة الاعتماديات غير المستخدمة (Tree Shaking) وتقليل حجم الأصول.
* لا تُطبق هذه الاستراتيجيات بشكل أعمى، بل بناءً على تحليل فعلي لاحتياجات المشروع وحجمه.

## 30. هندسة التعدد اللغوي والتوطين الثقافي (i18n & Localization Architecture)
* **شرط التفعيل:** يُفعّل هذا البند تلقائياً عندما يكون المشروع تطبيق ويب أو جوال يحتوي على واجهات مستخدم. يُعطّل تلقائياً للمشاريع الخلفية (Backend Only).
* يُلزم الوكيل بتصميم هيكلية المشروع من البداية لدعم التعدد اللغوي والتوسع العالمي، حتى لو بدأ المشروع بلغة واحدة. تغيير البنية لاحقاً مكلف جداً.

### أ. فصل النصوص عن الكود (Content Separation)
* يُحظر تماماً كتابة أي نص يظهر للمستخدم (Labels، رسائل خطأ، أزرار، عناوين) مباشرة في ملفات الكود (Hardcoded Strings).
* يُلزم الوكيل بتخزين جميع النصوص في ملفات JSON منفصلة داخل مجلد `locales/` بالهيكلية التالية:
  ```
  locales/
  ├── ar.json    ← العربية (افتراضي)
  ├── en.json    ← الإنجليزية
  └── [xx.json]  ← أي لغة مستقبلية (ملف واحد فقط للإضافة)
  ```
* يُستخدم نظام مفاتيح منظم بالتسلسل الهرمي (Nested Keys)، مثال:
  ```json
  {
    "auth": {
      "login": "تسجيل الدخول",
      "logout": "تسجيل الخروج",
      "welcome": "مرحباً، {{name}}"
    }
  }
  ```
* يُستخدم إطار i18n المناسب للتقنية المختارة (مثل `next-intl` أو `react-i18next` أو `vue-i18n` أو ما يعادلها).

### ب. دعم اتجاه الواجهة الديناميكي (RTL/LTR)
* يُلزم الوكيل بضبط اتجاه الصفحة ديناميكياً عبر خاصيتي `dir` و `lang` في عنصر `<html>` بناءً على اللغة النشطة:
  * العربية: `<html dir="rtl" lang="ar">`
  * الإنجليزية: `<html dir="ltr" lang="en">`
* يُستخدم الخصائص المنطقية (Logical Properties) في CSS بدلاً من الاتجاهية الثابتة:

  | ❌ ثابت (لا يدعم RTL) | ✅ منطقي (يدعم RTL تلقائياً) |
  |:---------------------|:--------------------------|
  | `margin-left` | `margin-inline-start` |
  | `margin-right` | `margin-inline-end` |
  | `padding-left` | `padding-inline-start` |
  | `text-align: left` | `text-align: start` |
  | `float: left` | `float: inline-start` |

### ج. التوطين الثقافي الكامل (Cultural Localization)
* يُلزم الوكيل باستخدام واجهات `Intl` المدمجة في JavaScript لعرض البيانات الحساسة ثقافياً حسب المنطقة (Locale) بدلاً من التنسيق اليدوي:
  * **العملات:** `Intl.NumberFormat(locale, { style: 'currency', currency })` — يعرض ر.س / $ / € / ¥ تلقائياً.
  * **التواريخ:** `Intl.DateTimeFormat(locale)` — يعرض التنسيق المحلي (هجري/ميلادي) تلقائياً.
  * **الأرقام:** `Intl.NumberFormat(locale)` — يعرض الفواصل والنقاط حسب المنطقة تلقائياً.
* يُحظر بناء دوال تنسيق مخصصة (Custom Formatters) لهذه الأغراض — `Intl` API يدعم جميع مناطق العالم أصلاً (البند 21 — `rules_code_quality.md`).

### د. أتمتة الترجمة (Translation Automation)
* عند إضافة نصوص جديدة في ملف اللغة الأساسية (`ar.json`)، يُلزم الوكيل بترجمتها تلقائياً إلى جميع ملفات اللغات الأخرى المعتمدة في المشروع.
* الترجمة التلقائية تُعتبر **مسودة أولية** وليست نهائية — يُنبّه الوكيل المستخدم لمراجعة الترجمات خاصة للمصطلحات التقنية أو التسويقية.
